# 顾客端号牌方案

## 目标

顾客可以通过扫描号牌二维码、NFC 链接，或进入通用顾客入口后输入号牌编号，查看自己以及同行人员的当前开台信息。方案需要兼顾商家低成本、灵活开台、多人同行、多人团购和顾客易用性。

## 核心结论

采用“一个顾客一个号牌，同行用 `group_id` 关联”的模型。

号牌二维码不绑定订单，也不绑定桌号，而是绑定一个可复用的顾客识别编号。开台时店员给每位顾客分配一个号牌编号，系统把该编号绑定到对应订单。多人同行订单共享同一个 `group_id`。顾客扫描同行任意一个号牌，都先找到自己的当前订单，再通过 `group_id` 查出同行所有未结账订单。

## 业务规则

1. 每位顾客对应一个号牌编号，例如 `1`、`18`、`A01`、`小熊`。
2. 商家可以预先批量生成一批号牌二维码或 NFC 链接，反复使用。
3. 固定号牌二维码只负责省略“输入编号”的步骤，本质仍是通用顾客入口。
4. 开台时如果人数为 `N`，页面展示 `N` 个号牌输入框。
5. 选择多人团购时，若团购配置 `persons = N`，开台人数自动调整为 `N`，号牌输入框也自动变为 `N` 个。
6. 扫描同行任意一个号牌，都展示整组订单。
7. 单人订单也允许拥有独立 `group_id`，但第一版可以只在多人时创建 `group_id`，顾客端兼容两种情况。
8. 同一个商家内，一个号牌编号同一时间只能绑定一个未结账订单。
9. 结账后该号牌自动释放，二维码/NFC 可复用。
10. 顾客端第一版只读，不允许顾客直接修改订单。

## 防误操作策略

前端人数输入设置：

- 默认人数：`1`
- 软上限：`8`
- 硬上限：`20`

当人数超过软上限时弹出确认提示，避免误输入大数字后生成大量输入框。当人数超过硬上限时直接阻止开台。

后端也必须保留硬上限校验，不能只依赖前端。

## 数据结构

### 顾客号牌表 `customer_access_codes`

| 字段 | 说明 |
| --- | --- |
| `code_id` | 主键 |
| `shop_id` | 商家 ID |
| `label` | 号牌编号，例如 `18`、`A01` |
| `token` | 二维码/NFC 中使用的永久随机 token |
| `is_active` | 是否启用 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

唯一约束：

- `shop_id + label` 唯一
- `token` 全局唯一

### 订单表新增字段

| 字段 | 说明 |
| --- | --- |
| `customer_code_id` | 绑定的号牌 ID，可为空 |
| `customer_code_label` | 绑定的号牌编号快照 |

现有 `phone` 字段继续作为收银台显示标识使用，避免大改前端。

## 顾客端入口

### 扫号牌二维码/NFC

```text
/customer.html?t=<token>
```

流程：

1. token 找到 `customer_access_codes`
2. 找到该号牌当前未结账订单
3. 通过订单 `group_id` 找同组订单
4. 返回顾客可见信息

### 通用入口

```text
/customer.html?shop=<shop_code>
```

顾客输入号牌编号后查询：

```text
GET /api/customer/by-code?shop_code=<shop_code>&label=<label>
```

## API 第一版

### 商家端

```text
GET  /api/customer-codes
POST /api/customer-codes/batch
```

`batch` 用于一次生成 `1-50` 或一组自定义编号，返回可打印链接列表。

### 顾客端

```text
GET /api/customer/by-token/{token}
GET /api/customer/by-code?shop_code=<shop_code>&label=<label>
```

返回结构：

```json
{
  "success": true,
  "shop": {"name": "TimerPro"},
  "access": {"label": "18"},
  "group_id": "xxx",
  "orders": [],
  "summary": {
    "count": 2,
    "estimated_total": 66.0
  }
}
```

## 开台交互

### 普通多人开台

店员输入人数 `3`，系统显示：

```text
号牌 1: [18]
号牌 2: [19]
号牌 3: [20]
```

提交后创建三条订单，并写入同一个 `group_id`。

### 多人团购开台

团购配置 `persons = 2`，店员选择该团购后自动显示两个号牌输入框。

```text
号牌 1: [18]
号牌 2: [19]
```

提交后两条订单共享同一个团购和 `group_id`。

## 2026-04-29 最新补充

通用顾客入口必须是“门店专属通用入口”，链接中自带 `shop_code`。顾客扫描所在门店的通用二维码后，只输入自己的号牌/标识，不再手动输入门店编号。

商家端 `GET /api/customer-codes` 和 `POST /api/customer-codes/batch` 会返回 `universal_url`，用于制作本店通用二维码：

```text
/customer.html?shop=<shop_code>
```

固定号牌/NFC 直达码仍保留：

```text
/customer.html?t=<token>
```

二者底层都查询当前门店内的有效未结订单，并通过 `group_id` 展示同行订单。

### 号牌生成与二维码导出

商家端设置页的号牌码管理已移入弹窗，避免设置页被长列表撑开。弹窗内支持：

- 连续编号生成：前缀 + 起始/结束数字
- 自定义号牌生成：一行一个，或用逗号/空格分隔，例如 `A01`、`小熊`、`VIP-1`
- 下载二维码：可选择是否在底部附带入口/号牌标识
- 下载 NFC 文件：后端生成 ZIP，每个入口/号牌各一个 `.txt`，文件内容就是写入 NFC 标签的 URL
- 下载全部物料包：包含 `qrcodes/` 与 `nfc/` 两个目录

二维码导出接口：

```text
GET /api/customer-codes/export
```

参数：

```text
scope=all|universal|badges
asset=bundle|qr|nfc
include_label=0|1
```

依赖：

```text
qrcode[pil]==8.2
```
